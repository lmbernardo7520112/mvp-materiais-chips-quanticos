## Question 1: What can be safely converted to SI?

### RSE (Research Software Engineer)

Physical constants (q, k_B, ε₀) are trivially safe — they are exact or
well-characterized. ε_r(Si) = 11.7 is a well-known T2 value that can be
introduced as a module constant without risk, **provided** the mode is
explicitly labeled and the default remains demonstrative.

**Verdict:** q, ε₀, ε_r(Si), ε_r(SiO₂) are safe for SI introduction.

### CMS (Computational Materials Scientist)

Agreed. The dielectric constants are among the most reliably known
semiconductor properties. D_it ranges are well-documented in review
literature (Sze & Ng 2007, Fleetwood 2018), but converting D_it to SI
requires specifying the energy integration convention, which is
non-trivial for a demonstrative model.

**Verdict:** ε_r is safe. D_it requires careful integration convention.

### SDP (Semiconductor Device Physicist)

I concur on ε_r. For D_it, the standard reporting unit (eV⁻¹ cm⁻²) is
universal in the field. Converting to J⁻¹ m⁻² is mathematically trivial
but physically meaningful only when paired with an energy window ΔE over
which the trap density is integrated. Without specifying ΔE, the surface
charge σ_eff = q · D_it · ΔE is undefined.

**Verdict:** D_it conversion is safe only with explicit ΔE convention.

### NMS (Numerical Methods Specialist)

From a numerical perspective, introducing ε = ε_r · ε₀ ≈ 10⁻¹⁰ changes
the Poisson equation coefficient by 10 orders of magnitude. The current
Jacobi solver should handle this (it divides by ε, so φ scales inversely),
but convergence criteria, tolerance, and max_iter may need adjustment.

**Verdict:** Safe if solver tolerance is scaled appropriately.

### GOV (Model Governance Specialist)

The key governance criterion is: does the conversion create an inference
pathway that a reader could mistake for calibration? For ε_r, the answer
is no — it is a well-known material property, not a fitted parameter.
For D_it with explicit ΔE, the answer is "only if labeled correctly."

**Verdict:** Safe with explicit labeling and overclaim gates.

## Question 3: What is the greatest overclaim risk?

### RSE

Producing φ in volts when the charge source is demonstrative. A reader
seeing "φ = 3.2 mV" might interpret this as a device prediction.

### CMS

Mixing literature ε_r with toy ρ_eff. The result has the right units
but the wrong magnitude — it is dimensionally correct but physically
meaningless.

### SDP

Using D_it without specifying the energy window ΔE. In real devices,
D_it is energy-dependent: D_it(E). Integrating over the full bandgap
vs a narrow window near mid-gap gives orders-of-magnitude different
surface charge.

### NMS

Numerical convergence at ε ~ 10⁻¹⁰ does not guarantee physical accuracy.
The solver could converge to a numerically stable solution that has no
physical relevance.

### GOV

**The greatest single overclaim risk is: presenting literature-scaled φ
as if it validates the model.** Literature-scaled mode proves dimensional
consistency, not physical accuracy.

## Council Decision

> [!IMPORTANT]
> The Research Council **recommends Option B** (literature-scaled constants
> only) as the probable first implementation step, conditional on:
>
> 1. ADR-008 being formally Accepted via PR review.
> 2. All pre-implementation gates (PG-1 through PG-5) being satisfied.
> 3. A dedicated implementation PR (not this ADR proposal).

### Explicit Prohibitions

The Council explicitly **prohibits** the following in any v0.4.x release:

1. **Do NOT proceed to v0.5.** The scale audit must be completed within v0.4.
2. **Do NOT implement Schrödinger.** Requires a separate ADR (v0.5+).
3. **Do NOT implement Option C directly.** Option B must be validated first.
4. **Do NOT implement nonlinear Poisson.** Beyond demonstrative scope.
5. **Do NOT implement self-consistent trap occupancy.** Beyond scope.
6. **Do NOT claim calibration.** Literature-scaled ≠ calibrated.

### Roadmap

```
v0.4.2 (current) → ADR-008 Proposed (documentation only)
v0.4.3 (future)  → ADR-008 Accepted + Option B implementation
v0.4.4 (future)  → Option C preparation (ΔE, t_eff, domain decision)
v0.5.0 (future)  → Option C implementation OR Schrödinger scope (new ADR)
```

## Technical Dissent / Required Corrections

The following corrections were identified during scientific hardening review
and are mandatory before the council approval is unconditional:

1. **N_ref_to_Dit_mapping expression was dimensionally incorrect.** The original
   expression `q · D_it · t_eff` conflated the surface charge mapping
   (q · D_it_SI · ΔE → σ_eff) with the volume conversion (σ_eff / t_eff → ρ_eff).
   t_eff only appears in the σ_eff → ρ_eff step, never in the D_it → σ_eff step.
   Corrected to two separate mappings: `D_it_to_sigma_mapping` and
   `sigma_to_rho_mapping`.

2. **The original φ scale audit missed the D_it magnitude factor.** The estimate
   used D_it_SI ~ 10²² (without the ΔE factor properly accounted), producing
   σ_eff ~ 10⁻¹⁶ C/m² and φ ~ 10⁻¹¹ V. The correct calculation with
   D_it_SI ≈ 6.242×10³³ J⁻¹m⁻² and ΔE = 1 eV yields σ_eff ≈ 1.6×10⁻⁴ C/m²
   and φ_scale ~ 10 V, which is drastically different and intentionally alarming.

3. **Option B must be treated as dimensional scaffolding, not physical output
   interpretation.** Substituting ε_r · ε₀ alone does not make φ physically
   interpretable. If δρ_eff and domain geometry remain demonstrative, φ must
   still be labeled demonstrative.

4. **Scale is highly sensitive, not "extremely small."** φ depends on ΔE (which
   can vary by 10³ between 1 meV and 1 eV), t_eff, L, and whether
   mean-subtraction is applied. No single φ estimate should be treated as
   representative without specifying all parameters.

**Council approval is conditional on these corrections being applied.**
All corrections have been applied in this hardened revision.

## Dissent Record

No fundamental dissent was recorded on the Option B recommendation. All five
council members voted unanimously for Option B as the recommended first
implementation step and agreed on the gate framework.

The only point of discussion was whether Option C should be attempted
within v0.4.x or deferred to v0.5. The consensus was:

- **CMS and SDP** favored deferring Option C to v0.5, arguing that
  the energy integration convention (ΔE) deserves its own ADR.
- **RSE and NMS** were neutral on timing but agreed that Option B
  must be validated before Option C.
- **GOV** recommended explicit language: "Option C is roadmap, not
  commitment."

**Post-hardening note:** The Technical Dissent section above documents
three dimensional corrections that were required before council approval
became unconditional. These corrections do not change the Option B
recommendation but strengthen its scientific rigor.

**Final consensus:** Option C is a roadmap item, not a v0.4.x commitment.
