# v0.7.0 C2 Interface Source / Regularization Council

## Date

2026-05-23

## Purpose

Deliberate on how σ_eff [C/m²] should be represented for future
electrostatic coupling. Evaluate five options (C2-A through C2-E)
and produce a recommendation.

## Question 1: Has the literature resolved a universal t_eff?

**Device Physicist:** No. Interface trap profiles are process-dependent.
The concept of a single t_eff is an oversimplification. Different
measurement techniques (charge pumping, conductance, DLTS) probe
different energy and spatial windows.

**Metrology Specialist:** No. D_it is an energy-resolved quantity
measured per unit area. Converting to volumetric density requires
assumptions about spatial distribution that vary with process,
anneal, and gate stack.

**TCAD Engineer:** In TCAD tools, interface charge is typically
implemented as a sheet charge or delta function at the interface.
Volume regularization is a mesh artifact, not physics.

**Conclusion:** No universal t_eff. C2-D rejected.

## Question 2: Should σ_eff be sheet source or volume regularized?

**PDE Specialist:** Sheet source (jump condition) is the mathematically
correct formulation for interface charge. It avoids introducing an
artificial length scale.

**Applied Mathematician:** For well-posed problems, the jump condition
n̂·(ε₂∇φ₂ − ε₁∇φ₁) = −σ is standard. Volume regularization is a
numerical convenience that must conserve charge.

**TCAD Engineer:** Most TCAD tools support both. Sheet charge is
cleaner but some FEM frameworks prefer volumetric source. Volume
regularization with l_reg ≈ 1–2 mesh cells is standard practice
for the latter case.

**Conclusion:** C2-A primary, C2-B fallback.

## Question 3: When is l_reg acceptable?

**PDE Specialist:** When the solver framework cannot support jump
conditions. l_reg should be ~1–2× the local mesh size. The result
must be insensitive to l_reg within a reasonable range.

**Applied Mathematician:** l_reg is acceptable if:
- charge conservation is verified: ∫ρ dV = ∫σ dA;
- results converge as l_reg → 0;
- l_reg is reported as numerical parameter, not physics.

**TCAD Engineer:** In practice, l_reg ≈ 0.1–1.0 nm for Si/SiO₂.
This is a mesh-dependent smearing, not a trap depth.

**Conclusion:** l_reg allowed as numerical parameter only. R1 tier.

## Question 4: Can literature/experiment inform depth priors?

**Metrology Specialist:** Yes, for sensitivity studies. Near-interface
traps (border traps) have been characterized at ~1–3 nm depth by
charge pumping and noise spectroscopy. But these are border traps,
not interface states proper.

**Device Physicist:** Literature can inform envelopes, not calibration.
Interface D_it is essentially at the interface (monolayer). Border
traps extend ~1–3 nm. Oxide traps are deeper. These three families
must not be confused.

**Conclusion:** C2-C conditional. Requires trap_family metadata.

## Question 5: How to avoid confusing interface traps with border/oxide traps?

**Metrology Specialist:** Require explicit trap_family declaration:
- interface → sheet charge (C2-A);
- border → near-interface depth prior (C2-C);
- oxide → bulk oxide, not relevant for C2.

**Device Physicist:** The C1 chain produces σ_eff from D_it (interface
states). This is inherently a sheet charge. Border/oxide traps are
a separate physical phenomenon with spatial extent.

**Conclusion:** Mandatory trap_family metadata for C2-C depth priors.

## Question 6: What must remain blocked?

**AI-RSE Auditor:** All implementation until ADR acceptance:
- ρ_eff API, t_eff parameter, l_reg code, solver coupling.

**Governance Auditor:** Additionally:
- C2-D/E (generic/calibrated t_eff);
- Quantum confinement solver;
- Calibration/coherence claims.

**Conclusion:** See ADR-012 "Still Blocked" section.

## Skeptical External Reviewer — Objections

### Objection 1: Sheet source is impractical for FEM

**Objection:** "Most finite-element frameworks don't natively support
delta-function source terms. You'll end up doing volume regularization
anyway."

**Response:** C2-B exists precisely for this case. The ADR designates
C2-A as the primary *physical* formulation and C2-B as the numerical
fallback. If the future solver requires C2-B, it must conserve charge
and report l_reg as metadata. The choice is solver-dependent, not
physics-dependent.

### Objection 2: l_reg is just t_eff with a different name

**Objection:** "Calling it l_reg instead of t_eff doesn't change the
physics. You're still dividing by a length."

**Response:** The distinction is semantic but critical for governance:
- l_reg is a numerical parameter tied to mesh resolution.
- t_eff implies a physical thickness with specific trap-depth meaning.
- l_reg should converge (result insensitive as l_reg → 0).
- t_eff implies a specific measurable quantity.
Conflating these invites false calibration.

### Objection 3: Border traps are important and you're ignoring them

**Objection:** "In real MOS QD devices, border traps at 1–3 nm are a
major source of 1/f noise. Ignoring spatial extent is unrealistic."

**Response:** C2-C explicitly allows literature/experimental depth
priors for border trap studies. The ADR does not ignore border traps;
it requires explicit trap_family declaration to prevent confusing
interface D_it profiles (monolayer) with spatially distributed border
trap populations. This is a scope delineation, not a denial.

### Objection 4: You'll never reach a real device model at this pace

**Objection:** "The governance overhead is disproportionate. You could
just implement ρ = σ/t and iterate."

**Response:** This is a scientific demonstrator, not a production TCAD
tool. The governance cost is justified because:
- Every parameter embedded now propagates to all future layers.
- A wrong t_eff becomes permanent technical debt.
- The ADR takes one documentation-only release; implementation follows
  immediately after acceptance.

### Objection 5: The scoring table is subjective

**Objection:** "Your option scoring uses arbitrary weights. Different
weights could produce different recommendations."

**Response:** The scoring is heuristic and transparent, not
authoritative. The unanimous council vote and physical arguments
are the basis for the recommendation, not the scoring alone.
The scoring serves as structured documentation, not as a decision
algorithm.

## Council Decision

**Unanimous recommendation:**

- **C2-A** — Interface sheet source / jump condition: **PRIMARY**
- **C2-B** — Conservative volume regularization: **FALLBACK**
- **C2-C** — Literature/experimental depth prior: **CONDITIONAL**
- **C2-D** — Generic physical t_eff: **REJECTED**
- **C2-E** — Calibrated device-fitted t_eff: **BLOCKED**

ADR-012 remains **Proposed**. Implementation requires acceptance
review and TDD RED phase.
