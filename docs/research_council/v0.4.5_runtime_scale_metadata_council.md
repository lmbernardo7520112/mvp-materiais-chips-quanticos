## Question 1: Is metadata-only runtime integration useful?

### RSE

Yes. Self-describing artifacts (CSVs, figures) are a best practice in
reproducible research software. If a CSV declares `scale_mode=demonstrative`,
any downstream consumer knows the numbers are not physically calibrated. This
is operationally valuable even without changing equations.

### CMS

Useful but not urgent. The materials science community expects clear
labeling of simulation regimes. Metadata in outputs reduces the risk
of someone mistaking demonstrative voltages for DFT-calibrated predictions.

### SDP

Strongly supportive. In semiconductor device physics, the distinction
between normalized and physical units is critical. A CSV that says
`phi_unit=dimensionless_demonstrative` prevents misuse in device modeling
workflows.

### NMS

Useful from a dimensional analysis perspective. Every numerical output
should carry its unit context. The metadata approach preserves numerical
identity (bitwise same phi) while enriching traceability.

### SMGS

Essential for model governance. The V&V chain requires that every artifact
state its operating regime. Without metadata, the burden falls on the
reader to infer the scale context from documentation — which is fragile.

## Question 2: What can be declared without altering physics?

### RSE

Scale mode, geometry mode, source mode, phi interpretation, phi unit label,
physical_interpretation_allowed flag, option_c_enabled flag, and a
literature_scaled_constants_available flag. All are string/boolean metadata
that do not enter any equation.

### CMS

Material identity (e.g., "Si" as the substrate), reference permittivity
value (for informational display only — not wired to epsilon), and
the evidence tier of each parameter used. None of these alter numerics.

### SDP

The presence or absence of interface trap conversion infrastructure
(D_it → σ_eff) can be declared as `option_c_enabled = false` without
implementing it. This is a statement about capability, not computation.

### NMS

Grid metadata (nx, ny, dx, dy) is already exported. Adding scale context
is a natural extension. The key invariant is: `phi_new == phi_old` for
any test case, before and after metadata integration.

### SMGS

Any metadata that passes a "does it change a number?" test can be declared.
If the answer is no, it's safe. If any pathway exists where the metadata
could influence a calculation (even indirectly via conditional logic),
it must be blocked until tested.

## Question 3: What would remain prohibited?

All panelists agree on the following prohibitions:

| # | Prohibited Action | Rationale |
|---|-------------------|-----------|
| P1 | Replacing `epsilon = 1.0` with `epsilon_r * epsilon_0` | Changes solver numerics |
| P2 | Computing `D_it_SI` from `D_it` | Requires delta_E_window |
| P3 | Computing `sigma_eff` from `D_it_SI` | Requires charge sign convention |
| P4 | Computing `rho_eff` from physical charge density | Changes Poisson source term |
| P5 | Using `t_eff` as physical oxide thickness | Requires oxide model validation |
| P6 | Computing `delta_E_window` | Requires literature trap energy bounds |
| P7 | Changing phi values in any test case | Breaks baseline comparisons |
| P8 | Claiming "calibrated" or "physical" for any output | Overclaim violation |

## Question 4: Which option is safest — A, B, or C?

| Expert | Vote | Rationale |
|--------|------|-----------|
| RSE | **B** | Metadata-only is low-risk, high-traceability. Standard practice. |
| CMS | **B** | Labeling regime is essential; passive scaffolding (A) is incomplete. |
| SDP | **B** | Device physics requires explicit unit declarations. B achieves this safely. |
| NMS | **B** | Dimensional analysis context in outputs is good practice. |
| SMGS | **B** | Governance requires self-describing artifacts. B is the minimum viable governance step. |

**Unanimous: Option B recommended for future implementation.**

Option A is safe but leaves a governance gap. Option C is premature and
prohibited.

## Question 5: What tests would be mandatory before implementation?

### Agreed Test Categories

| # | Test Category | Purpose |
|---|---------------|---------|
| T1 | Bitwise phi identity | Prove `phi_with_metadata == phi_without_metadata` for all test cases |
| T2 | CSV schema compatibility | Prove existing CSV consumers are not broken by new columns |
| T3 | Metadata invariant | Prove `physical_interpretation_allowed == False` in every output |
| T4 | Option C guard | Prove `option_c_enabled == False` in every runtime path |
| T5 | Scale mode propagation | Prove metadata flows from config → solver → CSV → figure |
| T6 | No epsilon change | Prove epsilon value in solver is unchanged (1.0 or whatever default) |
| T7 | Docstring audit | Prove no metadata field docstring claims calibration |

### RSE Additional

- Integration tests with `generate_all_results.py` confirming 10 figures
  + 5 CSVs are bitwise identical (or identical up to metadata columns).
- Quality gate extension to verify metadata fields in CSV headers.

### SMGS Additional

- A "metadata-only smoke test" that parses every CSV produced and asserts
  the presence of scale context fields and the absence of physical claims.

## Question 6: What overclaim risks exist?

| # | Risk | Severity | Mitigation |
|---|------|----------|------------|
| OC1 | Metadata field `literature_scaled_constants_available = true` interpreted as "constants are applied" | High | Rename to `literature_scaled_constants_module_exists` or add disclaimer |
| OC2 | `scale_mode = demonstrative` confused with "demonstration of real physics" | Medium | Docstring: "demonstrative means toy parameters, not physically meaningful" |
| OC3 | CSV column `epsilon_r_Si = 11.7` next to `phi` values suggests phi is in Si units | High | Only include epsilon_r as informational if phi is labeled `dimensionless` |
| OC4 | Figure titles mentioning "SI constants available" misread as "SI-calibrated" | Medium | Use "SI constants scaffolding (not applied to solver)" |
| OC5 | Future developer wires units.py to solver without TDD/ADR | Critical | Quality gate: grep for `from .units import` in solver files |

---

## Council Decision

### Resolution

1. **v0.4.5 is documentation-only.** No code changes. No runtime integration.
2. **Option B is recommended** as the next implementation target (v0.4.6+).
3. **Option C remains prohibited** until a dedicated ADR amendment.
4. **Option A is the current de facto state** and is acceptable as interim.
5. **Future Option B implementation requires:**
   - TDD plan with RED specifications.
   - Bitwise phi identity tests.
   - CSV schema compatibility tests.
   - Quality gate extension for metadata invariants.
   - Council re-review before merge.

### Dissenting Opinions

None. All panelists concur.
