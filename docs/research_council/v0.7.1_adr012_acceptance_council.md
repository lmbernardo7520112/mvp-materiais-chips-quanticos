## Purpose

## Question 1: Is ADR-012 physically mature?

**Device Physicist:** Yes. The jump condition n̂·(ε₂∇φ₂ − ε₁∇φ₁) = −σ_eff
is textbook MOS electrostatics. The five-option classification is
thorough and well-motivated.

**Metrology Specialist:** Yes. The distinction between interface
states, border traps, and oxide traps is correctly documented. The
metadata requirements for depth priors are comprehensive.

**TCAD Engineer:** Yes. Both C2-A (interface BC) and C2-B (regularized
volume) are standard TCAD approaches. The ADR correctly identifies
C2-A as physically faithful and C2-B as numerical fallback.

**PDE Specialist:** Yes. The conservation requirement ∫ρ dV = ∫σ dA
is the correct constraint for regularization. The ADR is mature.

**Conclusion:** Physically mature. Ready for acceptance.

## Question 2: Should C2-A be primary?

**All panelists:** Yes. Interface charge from D_it is inherently a
sheet charge. The jump condition is the natural formulation.

**Vote:** 9/9 — C2-A primary.

## Question 3: Should C2-B be fallback only?

**TCAD Engineer:** Yes. Volume regularization is only needed if the
solver framework cannot support jump conditions. It should not be
the default.

**PDE Specialist:** Agreed. The regularized form introduces l_reg
as an artificial parameter. Fallback is the correct role.

**Vote:** 9/9 — C2-B fallback only.

## Question 4: Should C2-C be conditional only?

**Metrology Specialist:** Yes. Depth priors from literature are
informative for sensitivity studies but must not be treated as
calibration. The 12-key metadata requirement is appropriate.

**Vote:** 9/9 — C2-C conditional.

## Question 5: Should generic physical t_eff remain rejected?

**Device Physicist:** Yes. There is no universal t_eff for Si/SiO₂
interface traps. Embedding one would create a false parameter.

**Applied Mathematician:** Agreed. The distinction between numerical
l_reg and physical t_eff is critical and must be maintained.

**Vote:** 9/9 — C2-D remains rejected.

## Question 6: Should C2 implementation remain blocked until RED?

**AI-RSE Auditor:** Yes. The TDD discipline requires RED tests
before implementation. ADR acceptance authorizes planning, not code.

**Governance Auditor:** Agreed. The v0.7.1 → v0.7.2 → v0.7.3
sequence (acceptance → RED → GREEN) is the correct workflow.

**Vote:** 9/9 — Implementation blocked until v0.7.2 RED.

## Question 7: What risks remain?

**All panelists identified ongoing risks:**

1. l_reg drifting into physical interpretation (R2).
2. C2-B becoming default out of convenience (new risk).
3. Depth priors treated as calibration (R6).
4. Solver coupling entering before separate ADR (R8).
5. Over-governance delaying the roadmap (R10).

**Mitigation:** All risks have documented mitigations in the risk
matrix. Future RED tests (15 candidates) address risks 1–4.
Risk 5 is mitigated by the single-release acceptance process.

## Skeptical External Reviewer — Objections

### Objection 1: l_reg will become physical t_eff in disguise

**Objection:** "Once you implement C2-B, someone will treat l_reg
as if it has physical meaning. The naming distinction is fragile."

**Response:** Future RED must include `test_l_reg_required_for_volume_regularization`
and sensitivity tests. Metadata must report l_reg as
`parameter_type = "numerical_regularization"`. Code review must
reject any parameter named t_eff in the C2 mapping module.

### Objection 2: C2-B will become default by convenience

**Objection:** "Jump conditions are harder to implement than
ρ = σ/l. Developers will default to C2-B even when C2-A is possible."

**Response:** The ADR explicitly designates C2-A as primary. Future
implementation must attempt C2-A first. C2-B is only authorized
when the solver framework cannot support jump/interface conditions.
This must be documented in the mapping module docstring.

### Objection 3: Depth priors will be treated as calibration

**Objection:** "Once you have depth numbers from literature, it's
a short step to claiming the model is calibrated."

**Response:** C2-C requires `calibration_status = "not_calibrated"`
in metadata. E1/E2 classification explicitly states "not calibration."
Future RED must include `test_e1_depth_prior_requires_metadata` and
`test_e2_depth_prior_requires_full_metadata`.

### Objection 4: Solver coupling will enter too early

**Objection:** "Accepting ADR-012 creates pressure to couple σ_eff
to the Poisson solver immediately."

**Response:** ADR-012 explicitly blocks solver coupling. The Still
Blocked list includes "Solver coupling" and "Electrostatic PDE
runtime coupling." Future solver coupling requires its own ADR
(per ADR-011 roadmap). C2 mapping produces source terms only;
it does not consume or invoke solver functions.

### Objection 5: Accepting without discrete equations is premature

**Objection:** "You're accepting a strategy without specifying the
discrete equations (finite difference stencils, mesh requirements).
The ADR is too abstract."

**Response:** ADR-012 is a *strategy* decision, not a numerical
specification. The discrete equations are the domain of the future
RED/GREEN implementation (v0.7.2/v0.7.3). The strategy decision
(sheet source vs volume regularization) is independent of the
discretization method. Accepting the strategy now does not commit
to any specific discretization scheme.
