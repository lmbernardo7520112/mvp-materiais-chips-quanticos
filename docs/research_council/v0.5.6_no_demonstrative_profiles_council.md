# v0.5.6 No Demonstrative Profiles Council

**Date:** 2026-05-15
**Subject:** Should E0 demonstrative profiles be rejected as operational evidence?
**Quorum:** 7 specialists

## Deliberation

### 1. Semiconductor Device Physicist

> E0 demonstrative profiles served their purpose during the proposal phase
> (v0.5.3). Now that ADR-010 is accepted with a clear evidence taxonomy,
> allowing E0 as an operational level creates confusion. A user running the
> model with an E0 profile might present results as physically meaningful
> when they are not.
>
> S0 TEST_ONLY solves the testing need without contaminating the scientific
> output space.
>
> **E0 operational?** Reject.
> **S0 test-only?** Accept — essential for verification.
> **E1 minimum?** Yes.
> **Risks?** None if S0 is properly confined to tests.
> **Vote:** Accept amendment.

### 2. Interface Trap Metrology Specialist

> Measurement techniques produce D_it(E) profiles with specific physical
> meaning. Allowing demonstrative (invented) profiles alongside measured
> ones muddies the epistemic status of the model output. E1 as the floor
> for operational profiles is appropriate: even a coarse literature estimate
> carries more physical meaning than a synthetic curve.
>
> **E0 operational?** Reject.
> **E1 minimum?** Correct — literature at least provides order-of-magnitude
> grounding.
> **E2 conditional?** Yes, with source metadata.
> **Vote:** Accept amendment.

### 3. Silicon Spin Qubit / Charge Noise Specialist

> For qubit-relevant modeling, the D_it profile near the Fermi level
> matters physically. A demonstrative profile with arbitrary values in
> that region could mislead about noise expectations. I support making
> E1 the minimum.
>
> **E0 operational?** Reject.
> **S0 test-only?** Accept.
> **Vote:** Accept amendment.

### 4. Computational Materials Scientist

> From a computational perspective, S0 test fixtures are essential for
> verifying numerical integration (convergence, bin-width sensitivity,
> backward compatibility). But they should be clearly separated from
> model-facing profiles. The S0/E1 distinction is clean and enforceable.
>
> **E0 operational?** Reject.
> **S0 test-only?** Accept.
> **E1 minimum?** Yes.
> **Vote:** Accept amendment.

### 5. Applied Mathematician / Numerical Verification Specialist

> Test fixtures with known analytical solutions are indispensable for
> convergence verification. A uniform profile (single bin) tests P0
> backward compatibility. A step function tests bin-edge handling. These
> must exist as S0 fixtures.
>
> However, these synthetic profiles must not appear in any output that
> could be interpreted as a model prediction. The S0 label and test-only
> restriction are necessary and sufficient.
>
> **E0 operational?** Reject.
> **S0 test-only?** Accept — required for mathematical rigor.
> **Risk of losing verification?** None, because S0 covers all test needs.
> **What v0.6.0 RED must enforce:** S0 fixtures in test files only; E1/E2
> evidence label on all model-facing profiles.
> **Vote:** Accept amendment.

### 6. Research Software Engineer

> This is a clean governance improvement. The current evidence taxonomy
> has five levels (E0–E4). After this amendment:
>
> - S0: test-only (not an evidence level)
> - E0: deprecated (not operational)
> - E1: minimum operational
> - E2: conditional
> - E3/E4: blocked
>
> The v0.6.0 RED phase should enforce this in code: every profile struct
> must carry an evidence_level field that rejects E0 at construction time
> for model-facing instances.
>
> **E0 operational?** Reject.
> **S0 test-only?** Accept.
> **Governance improvement?** Clear improvement — reduces ambiguity.
> **Vote:** Accept amendment.

### 7. Skeptical External Reviewer

> **Objection 1:** Deprecating E0 before any D_it(E) code exists is
> premature governance. You are restricting a feature that does not yet
> exist. This could delay v0.6.0 by adding unnecessary constraints.
>
> **Response:** The amendment is documentation-only and adds no code
> constraints today. It establishes intent before implementation, which is
> the correct governance sequence. The v0.6.0 RED tests will enforce E1
> minimum, but that enforcement is part of normal TDD, not additional
> overhead.
>
> **Objection 2:** S0 TEST_ONLY is functionally identical to E0
> demonstrative — you are just renaming it. The restriction is cosmetic.
>
> **Response:** The distinction is not cosmetic. E0 was defined as an
> evidence level (part of the E0–E4 taxonomy), implying it could label
> model outputs. S0 is explicitly NOT an evidence level — it is a test
> fixture category. S0 profiles cannot appear in CSV artifacts, benchmark
> tables, or release figures. E0 profiles could.
>
> **Objection 3:** What if a researcher wants to explore a hypothetical
> D_it(E) shape for sensitivity analysis? Forcing E1 means they must find
> a literature reference for every scenario.
>
> **Response:** Sensitivity analysis with arbitrary profiles is valuable
> but should be labeled as such. The amendment allows S0 for internal
> testing. If a researcher wants to publish sensitivity results, they
> should use E1-grounded ranges (e.g., "D_it varies from literature low
> to literature high") rather than invented profiles.
>
> **Vote:** Accept amendment, with the condition that the S0/E0
> distinction is clearly documented in the amendment text.

## Council Decision

1. **E0 demonstrative profiles are rejected** as operational evidence for
   C1 D_it(E) profiles.
2. **S0 TEST_ONLY** is introduced as a test fixture category (not an
   evidence level).
3. **E1 literature-informed** is the minimum operational evidence level.
4. **E2** remains conditional with metadata.
5. **E3/E4** remain blocked.
6. **v0.6.0 RED** must enforce evidence_level >= E1 for model-facing
   profiles and confine S0 to test suites.
7. **C2 remains blocked.** No ρ_eff, t_eff, solver coupling, or φ.
8. **Skeptical objections** are incorporated: S0/E0 distinction is
   documented in the ADR-010 amendment.
