## Context

The v0.4.0 release implemented the **Demonstrative Poisson Bridge Core** — a
Boussinesq-inspired linearized trap-charge closure (Option C-B from ADR-007)
coupled to a 2D Poisson solver with homogeneous permittivity and Dirichlet
boundary conditions. All fields (δρ_eff, φ) are demonstrative and uncalibrated.

The v0.4.1 release consolidated **deep parametric curation**:

- A 20-parameter registry with evidence tiers (CONST/T0/T1/T2/T3/TX/NUM).
- 11 literature sources audited (Sze, Bracht, Sinno, Watkins, Fleetwood,
  Connors, Zwanenburg, Burkard, Stesmans, CODATA, Ioffe).
- Unit audit: C/m² vs C/m³ documented; eV⁻¹ cm⁻² → J⁻¹ m⁻² documented.
- D0_def corrected: D0 vs D(T) separated.
- ε₀ corrected as CONST_DERIVED, not CONST_EXACT.
- N_ref_active separated from N_ref_to_Dit_mapping.
- Connors2022 reclassified as context_anchor.
- source_role / parameter_anchor / equation_anchor / context_anchor present.
- Zero changes to src/, scripts/, or tests/.

**The next risk is physical scale.**

The Poisson solver currently operates with ε = 1.0, a normalized domain
(x, y ∈ [0, 1]), and demonstrative charge magnitudes. Introducing SI units
(ε = ε_r · ε₀, physical D_it, real domain lengths) without rigorous
infrastructure would create **false calibration** — outputs that appear
physically meaningful but lack the dimensional, geometric, and scientific
grounding to support any physical interpretation.

**This ADR proposes only the scope and criteria for a future SI unit conversion
and parameter scale audit infrastructure. It does NOT authorize implementation.**

## Decision

Because ADR-008 is now **Accepted**, it approves the planning requirements
for a future SI unit conversion and scale-audit infrastructure. It still
**does not authorize implementation**. Any implementation requires a
dedicated PR satisfying the acceptance record above.

> No code-level unit conversion is authorized by this ADR. Implementation
> may only proceed via a separate PR that satisfies all 13 prerequisites
> listed in the Acceptance Record.

This ADR establishes:

1. The dimensional chain from D_it to φ.
2. The distinction between "demonstrative mode" and "literature-scaled mode."
3. The constraints inherited from ADR-007.
4. The future implementation requirements for any unit conversion.
5. The acceptance gates for this proposal.

## Proposed Dimensional Chain

The physically correct chain from interface trap density to electrostatic
potential is:

```
D_it [eV⁻¹ cm⁻²]
  → D_it_SI = D_it × (1 eV / 1.602×10⁻¹⁹ J) × (1 cm / 10⁻² m)⁻²
            = D_it × 6.242×10¹⁸ × 10⁴   [J⁻¹ m⁻²]
  → σ_eff = q · D_it_SI · ΔE   [C/m²]
      where ΔE is the energy integration window [J]
  → ρ_eff = σ_eff / t_eff   [C/m³]
      where t_eff is the effective interface thickness [m]
  → ∇·(ε₀ ε_r ∇φ) = −ρ_eff
  → φ   [V]
```

> [!IMPORTANT]
> This chain is **NOT implemented** in the current codebase.
> The v0.4 implementation uses demonstrative scales:
> - ε = 1.0 (not ε_r · ε₀)
> - N_ref = 1.0 (not q · D_it_SI · ΔE)
> - Domain is [0, 1] × [0, 1] (normalized, not physical geometry)
> - δρ_eff is mean-subtracted and uncalibrated
> - φ is demonstrative — its magnitude has no physical meaning

Any physical interpretation requires:
1. Explicit unit conversion at every step.
2. Physical domain geometry (Lx, Ly in meters).
3. Justified t_eff (not arbitrary).
4. Energy integration convention for D_it.
5. Calibration against known reference solutions.

## Accepted Constraints from ADR-007

The following constraints from ADR-007 remain binding:

1. **C_def remains dimensionless** (tier TX, adimensional proxy).
2. **δρ_eff remains demonstrative** — mean-subtracted perturbation, not
   calibrated physical charge density.
3. **Poisson solver is homogeneous** — ∇²φ = −δρ_eff/ε with uniform ε.
4. **Boundary conditions are Dirichlet** — demonstrative, not device geometry.
5. **No Poisson non-linear** — no self-consistent Poisson-Boltzmann.
6. **No self-consistency** — no φ → trap occupancy → ρ_eff → φ loop.
7. **No device-level prediction** — no chip performance claims.
8. **λ_T, λ_C remain T0** — uncalibrated sensitivity parameters.

## What Does "Demonstrative Mode" vs "Literature-Scaled Mode" Mean?

### Demonstrative Mode (current v0.4)

- All physical constants replaced by ε = 1.0, N_ref = 1.0, t_eff = 1.0.
- Domain normalized to [0, 1] × [0, 1].
- Output magnitudes are numerically convenient but physically meaningless.
- Purpose: verify numerical correctness, test pipeline integrity,
  demonstrate the process-to-device bridge concept.

### Literature-Scaled Mode (future, requires ADR-008 acceptance)

- Physical constants at SI values: ε = ε_r(Si) · ε₀.
- D_it converted to J⁻¹ m⁻² with explicit energy integration.
- Domain may be physical (Lx, Ly in nm or μm) or explicitly normalized.
- Output magnitudes must be checked against order-of-magnitude expectations.
- Purpose: verify that the model produces results in the correct physical
  range, without claiming calibration or device prediction.

> [!WARNING]
> Literature-scaled mode is **NOT** calibrated mode.
> Using literature values for ε_r and D_it does not constitute calibration.
> It merely ensures that the model operates in the correct order-of-magnitude
> regime for silicon/SiO₂ interfaces.

> [!IMPORTANT]
> **Option B — literature-scaled constants only — is safe only as dimensional
> scaffolding.** If δρ_eff and domain geometry remain demonstrative, φ must
> still be labeled demonstrative and must not be reported as volts with
> physical interpretation. Substituting ε_r · ε₀ alone does not make φ
> physically interpretable.

## Key Questions This ADR Must Resolve

### Q1–Q3: Parameter classification for SI conversion

1. **Which parameters can receive SI units without false calibration?**
   - q, ε₀ (physical constants — always safe)
   - ε_r(Si) = 11.7, ε_r(SiO₂) = 3.9 (well-established T2 — safe)

2. **Which parameters may use literature ranges as scale guides?**
   - D_it = 10¹⁰–10¹² eV⁻¹cm⁻² (range, not point value — safe if labeled)
   - D0_def literature range (safe as order-of-magnitude guide)

3. **Which parameters must remain toy/demonstrative?**
   - λ_T, λ_C (no physical anchor)
   - C_sat (arbitrary)
   - N_ref_active (explicitly TX)
   - t_eff (requires device-specific justification)

### Q4–Q6: Unit conversion specifics

4. **D_it conversion:** eV⁻¹ cm⁻² → J⁻¹ m⁻² requires:
   - Energy unit: ×(1/1.602×10⁻¹⁹) = ×6.242×10¹⁸
   - Area unit: ×(10²)² = ×10⁴
   - Combined: ×6.242×10²²
   - Plus explicit energy integration window ΔE.

5. **σ_eff → ρ_eff conversion:** σ_eff [C/m²] → ρ_eff = σ_eff / t_eff [C/m³]
   - Requires t_eff in meters.
   - Typical interface thickness: 0.5–5 nm (literature).
   - t_eff = 1 nm is common but not universal.
   - Using arbitrary t_eff creates false precision.

6. **ε replacement:** ε = 1.0 → ε = ε_r · ε₀
   - For Si: ε = 11.7 × 8.854×10⁻¹² ≈ 1.036×10⁻¹⁰ F/m
   - This changes φ magnitude by ~10¹⁰ relative to ε = 1.0.
   - The solver equation ∇²φ = −ρ_eff/ε would produce completely
     different numerical values.

### Q7–Q9: Domain geometry

7. **Is the current domain physical or normalized?**
   - Normalized: x, y ∈ [0, 1], dx = dy = 1/(N−1).
   - No physical length scale is attached.

8. **If physical, what would Lx, Ly be?**
   - Typical silicon quantum dot pitch: 50–200 nm.
   - Typical interface study domain: 100 nm – 10 μm.
   - Choice depends on the physical question being modeled.

9. **If normalized, how to prevent physical misinterpretation?**
   - Explicit labels: "normalized coordinates (dimensionless)."
   - Output labeled "demonstrative φ (not in volts)."
   - Domain metadata: `is_physical: false, unit: "normalized"`.

### Q10–Q11: Sanity checks

10. **Order-of-magnitude audit for φ:**

    For D_it = 10¹¹ eV⁻¹ cm⁻²:

    ```
    D_it_SI = 10¹¹ × 6.242×10²² ≈ 6.242×10³³ J⁻¹ m⁻²
    ```

    If ΔE = 1 eV = 1.602×10⁻¹⁹ J:

    ```
    D_it_SI · ΔE ≈ 10¹⁵ m⁻²
    σ_eff ≈ q · 10¹⁵ ≈ 1.6×10⁻⁴ C/m²
    ```

    If t_eff = 1 nm = 10⁻⁹ m:

    ```
    ρ_eff ≈ σ_eff / t_eff ≈ 1.6×10⁵ C/m³
    ```

    If L = 100 nm = 10⁻⁷ m and ε ≈ 1×10⁻¹⁰ F/m:

    ```
    φ_scale ~ ρ_eff · L² / ε
            ~ 1.6×10⁵ × 10⁻¹⁴ / 10⁻¹⁰
            ~ 10 V
    ```

    > [!WARNING]
    > This example is intentionally alarming: φ is highly sensitive to ΔE,
    > D_it, t_eff, L, neutralization/mean-subtraction, boundary conditions,
    > and geometry. Therefore, literature-scaled mode must include scale
    > sanity checks and must not be interpreted as calibrated device
    > electrostatics.

    If ΔE = 1 meV instead of 1 eV, σ_eff, ρ_eff, and φ_scale are all
    reduced by 10³ (φ_scale ~ 10 mV).

    The scale is **not** "extremely small" — it depends entirely on the
    choice of ΔE, D_it, t_eff, L, and whether mean-subtraction or
    neutralization is applied.

11. **Required dimensional sanity checks:**
    - [ρ_eff] = C/m³ ✓
    - [ε∇²φ] = (F/m)(V/m²) = C/m³ ✓
    - [σ_eff / t_eff] = (C/m²) / m = C/m³ ✓
    - [q · D_it_SI · ΔE] = C · J⁻¹m⁻² · J = C/m² ✓

### Q12: Overclaim prevention

12. **How to prevent false calibration?**
    - Separate "demonstrative mode" (ε = 1.0) from "literature-scaled mode."
    - Both modes must be available; literature-scaled does NOT replace demonstrative.
    - All outputs in literature-scaled mode must carry explicit disclaimers.
    - No φ value may be presented as a device prediction.
    - Quality gates must check for forbidden calibration language.

### Q13–Q14: Governance

13. **Changes requiring policy.json update:**
    - Introducing a `v0.5` stage.
    - Adding new authorized files for SI conversion modules.
    - Changing forbidden term lists.

14. **Changes requiring new ADR:**
    - Implementing Schrödinger equation (ADR-009 or later).
    - Implementing nonlinear Poisson (new ADR).
    - Implementing self-consistent trap occupancy (new ADR).
    - Implementing TCAD/QTCAD integration (new ADR).
    - Claiming calibrated physical predictions (new ADR).

## Future Implementation Requirements

Any implementation of SI unit conversion (in a future PR, after ADR-008 is
accepted) **must** satisfy:

1. **Explicit unit layer:** A module or convention that makes every unit
   conversion traceable and testable.

2. **Preserve demonstrative mode:** The ε = 1.0, N_ref = 1.0 mode must
   remain functional and be the default.

3. **Add literature-scaled mode separately:** A new mode flag or parameter
   set, clearly labeled, that uses SI values.

4. **Document all conversions:** Every unit conversion must have a docstring
   or comment with the conversion formula and reference.

5. **Test dimensional consistency:** Unit tests that verify [ρ_eff] = C/m³,
   [φ] = V, [ε∇²φ] = C/m³ for the literature-scaled path.

6. **Test order of magnitude of φ:** For known D_it, ε_r, t_eff, L, the
   output φ must be within expected order-of-magnitude.

7. **Prevent overclaims:** Quality gates must scan for "calibrated,"
   "validated," "predicts," "device performance" in outputs and logs.

8. **Maintain quality gates:** All existing gates (pytest, coverage ≥ 70%,
   ruff, pyright, generate_all_results, AI-RSE GateOps) must pass.

## Non-Goals

The following are **explicitly out of scope** for this ADR and any
implementation it authorizes:

| Non-Goal | Rationale |
|----------|-----------|
| Experimental calibration | No experimental data available |
| Parameter fitting to real device data | No device data |
| Schrödinger equation | Quantum confinement not modeled |
| Nonlinear Poisson | Beyond demonstrative scope |
| Self-consistent trap occupancy | Requires iterative φ-ρ loop |
| TCAD/QTCAD integration | External tool dependency |
| Device-level prediction | Explicitly forbidden |
| Coherence/fidelity prediction | Requires spin Hamiltonian |
| Charge-noise prediction (real) | Requires fluctuator models |
| Gate fidelity prediction | Requires quantum control theory |
| Real chip performance claims | Explicitly forbidden |

## Consequences

### Positive

- Establishes a rigorous framework before any unit conversion is attempted.
- Prevents ad-hoc SI introduction that could create false calibration.
- Documents the full dimensional chain for future implementers.
- Enables informed decision-making about demonstrative vs literature-scaled modes.
- Maintains scientific integrity of the project.

### Negative

- Delays SI unit conversion to a future release.
- Adds governance overhead (another ADR, more documentation).
- May be perceived as excessive caution by external readers.

### Mitigations

- The delay is justified by the critical risk of false calibration.
- The documentation serves as a design specification for future work.
- The project's governance standards are a feature, not a bug.
