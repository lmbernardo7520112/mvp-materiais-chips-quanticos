# Decision Brief — v0.4.5 Runtime Scale Metadata Integration

> **Date:** 2026-05-12
> **Author:** Staff Research Software Engineer (AI-assisted)
> **Status:** PROPOSED
> **ADR Reference:** ADR-008 (Accepted in v0.4.3)
> **Preceding Release:** v0.4.4 — SI Constants Scaffolding

---

## Context

The v0.4.4 release introduced two passive infrastructure modules:

- `units.py` — SI physical constants (ε₀, q_e) and permittivity lookups.
- `scale_modes.py` — Scale mode, geometry mode, and potential interpretation metadata.

These modules are fully tested (31 tests, all passing) and available at import
time, but **no runtime code uses them**. The Poisson bridge solver, CLI scripts,
and CSV/figure generators remain unaware of the scale metadata infrastructure.

This decision brief evaluates three integration strategies for v0.4.5+.

> **Important:** This v0.4.5 release is documentation-only. No implementation
> occurs. The decision brief records the analysis; implementation is deferred
> to a future release (v0.4.6+) pending council approval and TDD planning.

---

## Option A — Keep Scaffolding Passive

### Description

- `units.py` and `scale_modes.py` remain importable but unused at runtime.
- No CSV fields change.
- No script changes.
- No solver changes.
- Minimum risk of overclaim.
- Minimum operational utility.

### Advantages

| # | Advantage |
|---|-----------|
| A1 | Maximum scientific safety — zero chance of altering results. |
| A2 | Zero risk of metadata being misinterpreted as calibration. |
| A3 | Preserves v0.4.4 as a pure foundation layer. |
| A4 | No test regression risk. |
| A5 | Simplest governance posture. |

### Disadvantages

| # | Disadvantage |
|---|--------------|
| A6 | Metadata does not appear in artifacts — scale context is invisible. |
| A7 | CSV consumers cannot distinguish demonstrative from physical outputs. |
| A8 | No runtime evidence that `physical_interpretation_allowed = False`. |
| A9 | Delayed return on v0.4.4 investment. |

### Verdict

Safe but passive. Acceptable as interim state but does not advance
auditability.

---

## Option B — Metadata-Only Runtime Declaration

### Description

- Runtime passes `ScaleMetadata` to artefact generators.
- CSV outputs may include additional metadata columns or header comments:
  - `scale_mode = demonstrative`
  - `geometry_mode = normalized_2d`
  - `physical_interpretation_allowed = false`
  - `literature_scaled_constants_available = true`
  - `option_c_enabled = false`
- **No equation changes.**
- **No epsilon substitution.**
- **No D_it coupling.**
- **No σ_eff or ρ_eff physical computation.**
- **phi values remain identical.**

### Advantages

| # | Advantage |
|---|-----------|
| B1 | Improves auditability — every CSV self-declares its scale context. |
| B2 | Makes explicit that φ is demonstrative, not physical. |
| B3 | Prepares future evolution without altering mathematics. |
| B4 | Enables automated checks: "was this output generated in demonstrative mode?" |
| B5 | Aligns with ADR-008 Option B scaffolding intent. |

### Disadvantages / Risks

| # | Risk |
|---|------|
| B6 | User may misinterpret metadata fields as endorsement of physical accuracy. |
| B7 | Requires new tests to verify metadata propagation and no numerical change. |
| B8 | Requires careful CSV format design to avoid breaking existing consumers. |
| B9 | Must enforce `physical_interpretation_allowed = False` as invariant. |

### Implementation Requirements (Future)

1. TDD plan with RED specifications before any code.
2. Non-regression tests proving `phi` values are bitwise identical.
3. CSV schema versioning or backward-compatible extension.
4. Explicit docstring disclaimers on every metadata field.
5. Quality gate extension to verify metadata invariants.

### Verdict

**Recommended as next implementation target (v0.4.6+).** Not implemented in
v0.4.5. This release only documents the decision and specifies acceptance
criteria.

---

## Option C — Numerical Coupling to Solver

### Description

- Replace `epsilon = 1.0` with `epsilon_r · epsilon_0` in the Poisson solver.
- Wire `units.py` constants directly into `poisson_solver_2d.py`.
- Compute `D_it_SI` from `D_it` via energy integration.
- Derive `σ_eff` and `ρ_eff` from physical charge densities.
- Apply `t_eff` as physical oxide thickness.
- Introduce `delta_E_window` for trap energy integration bounds.

### Status

> **🚫 PROHIBITED in v0.4.5 and any release without dedicated ADR amendment.**

### Rationale for Prohibition

| # | Reason |
|---|--------|
| C1 | Changes solver numerics — requires full regression validation. |
| C2 | φ values would change, breaking existing baseline comparisons. |
| C3 | Risk of false physical interpretation without calibration. |
| C4 | Requires charge sign convention decisions not yet made. |
| C5 | Requires delta_E_window justification from literature. |
| C6 | Requires t_eff physical basis (not yet established). |
| C7 | Violates ADR-008 phasing: Option C after Option B validation. |

### Prerequisites for Future Consideration

1. ADR-008 Amendment or new ADR explicitly authorizing Option C scope.
2. Dedicated TDD plan with perturbation-mode sanity checks.
3. Council re-deliberation with explicit risk acceptance.
4. Non-regression test proving Option B metadata is stable.
5. Literature evidence for delta_E_window bounds.
6. Physical charge sign convention documented and tested.

## Recommendation

| Option | v0.4.5 Decision |
|--------|-----------------|
| **A** | Acceptable as current state. No action required. |
| **B** | **Recommended** as future v0.4.6 implementation target. Specified in this document. |
| **C** | **Prohibited** until new ADR. |

The v0.4.5 release formalizes the decision to pursue Option B in a future
release, documents the acceptance criteria, and blocks Option C.
