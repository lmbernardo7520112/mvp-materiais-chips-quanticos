# v0.7.0 C2 Interface Source / Regularization Risk Matrix

## Purpose

Identify and mitigate risks for the C2 decision and future implementation.

---

### R1 — Invented physical t_eff

- **Severity:** Critical
- **Failure mode:** A generic thickness parameter is embedded as
  physical t_eff, creating false calibration.
- **Mitigation:** ADR-012 rejects C2-D. Code reviews must reject
  any parameter named t_eff in C2 mapping code.
- **Responsible skill:** physics-dimensional-audit, scope-guardrails
- **Release gate:** Future RED must include `test_generic_t_eff_rejected`.

### R2 — Numerical l_reg mistaken as physical thickness

- **Severity:** Critical
- **Failure mode:** l_reg is documented or used as if it represents
  physical trap depth.
- **Mitigation:** Metadata must declare l_reg as "numerical_regularization".
  Tests must verify l_reg is reported, not hidden.
- **Responsible skill:** physics-dimensional-audit, report-auditor
- **Release gate:** Future RED must include sensitivity test to l_reg.

### R3 — Interface traps confused with border/oxide traps

- **Severity:** Critical
- **Failure mode:** Border trap depth priors are applied to D_it
  interface-state profiles without declaring trap_family.
- **Mitigation:** C2-C requires explicit trap_family metadata.
  Interface D_it → sheet charge (C2-A). Border traps → C2-C only.
- **Responsible skill:** scope-guardrails
- **Release gate:** Future RED must include `test_interface_vs_border_trap_family_required`.

### R4 — Volume regularization violates charge conservation

- **Severity:** Critical
- **Failure mode:** ∫ρ dV ≠ ∫σ dA after regularization.
- **Mitigation:** Conservation equation documented in ADR-012.
  Future tests must verify integral equality.
- **Responsible skill:** physics-dimensional-audit
- **Release gate:** Future RED must include `test_integral_rho_matches_integral_sigma`.

### R5 — Future potential depends pathologically on l_reg/grid

- **Severity:** High
- **Failure mode:** Electrostatic potential φ changes qualitatively
  with l_reg or mesh refinement.
- **Mitigation:** Require convergence study as part of future GREEN.
  l_reg sensitivity study in documentation.
- **Responsible skill:** physics-dimensional-audit
- **Release gate:** Future GREEN must demonstrate convergence.

### R6 — Literature depth prior treated as calibration

- **Severity:** Critical
- **Failure mode:** A border trap depth from literature is used as
  if it calibrates the MVP for a specific device.
- **Mitigation:** C2-C requires calibration_status = "not_calibrated".
  E1/E2 classification only. E3/E4 blocked.
- **Responsible skill:** report-auditor, scope-guardrails
- **Release gate:** Future RED must include `test_e1_depth_prior_requires_metadata`.

### R7 — C2 implemented before ADR acceptance

- **Severity:** Critical
- **Failure mode:** Code implementing σ → ρ mapping is committed
  before ADR-012 is formally accepted.
- **Mitigation:** ADR-012 status = Proposed. release-manager skill
  blocks merge of implementation before acceptance.
- **Responsible skill:** release-manager, ai-rse-gateops
- **Release gate:** ADR-012 must be Accepted before any C2 code.

### R8 — Solver coupling sneaks into C2 mapping

- **Severity:** Critical
- **Failure mode:** C2 mapping module imports or calls solver functions.
- **Mitigation:** Future RED must include `test_no_solver_import_in_mapping_module`.
  C2 mapping produces source terms only; solver consumes them separately.
- **Responsible skill:** scope-guardrails
- **Release gate:** Future RED test and policy gate.

### R9 — Future confinement prototype inherits artificial potential

- **Severity:** High
- **Failure mode:** A future quantum confinement solver uses φ
  computed from regularized ρ with arbitrary l_reg, producing
  meaningless confinement energies.
- **Mitigation:** ADR-012 requires convergence study. Future v0.9.0
  ADR must verify input potential quality before confinement.
- **Responsible skill:** physics-dimensional-audit
- **Release gate:** v0.9.0 ADR prerequisite.

### R10 — Roadmap stalls due to over-governance

- **Severity:** Medium
- **Failure mode:** Excessive documentation requirements delay
  implementation indefinitely.
- **Mitigation:** ADR-012 is a single documentation-only release.
  After acceptance, implementation proceeds via standard TDD.
  The ADR process adds ~1 release of documentation, not indefinite delay.
- **Responsible skill:** release-manager
- **Release gate:** v0.7.0 closes documentation; v0.7.1 opens RED.
