# v0.7.6 AI-for-Science Separation Architecture

## Purpose

Define how AI-for-Science may eventually coexist with the classical track without contaminating it.

No directory, module, or infrastructure is created in v0.7.6. This document is a future reference only.

## Proposed Future Directory Boundary

If ever implemented, future AI-for-Science code must live under a clearly isolated namespace:

**Option A:**
```
src/mvp_quantum_materials/ai_experimental/
```

**Option B:**
```
experiments/ai_for_science/
```

No such directory is created in v0.7.6. The choice between options will be made in a future ADR.

## Required Future Isolation

- No imports from classical modules (thermal_solver, diffusion_solver, poisson_solver_2d, c2_charge_mapping, surface_charge, effective_charge, energy_profiles, dit_profile_library) unless explicitly allowed by ADR.
- No replacement of classical solver functions.
- No modification of C1/C2/C3 modules.
- No project results mixed with AI-for-Science outputs without explicit metadata labeling.
- All AI-for-Science outputs must be labeled as `experimental` in metadata.
- ML dependencies must be in an optional dependency group (e.g., `[ml]` or `[ai-experimental]`), never in core requirements.
- CI matrix must include an ML-free run that validates the classical path independently.
- Stochastic reproducibility controls (fixed seeds, deterministic mode) are required in all AI-for-Science code.

## Integration Boundary

AI-for-Science outputs may only be **compared against** classical baselines.

They cannot become source of truth without:

- Future ADR acceptance.
- Benchmark evidence (analytic + classical solver).
- Validation with error quantification.
- Human approval recorded in decision log.

## Classical Track Guarantees

The following guarantees must hold even if an AI-for-Science track is eventually activated:

- `pytest` without ML optional dependencies must pass all existing tests.
- `generate_all_results.py` must produce identical outputs regardless of AI track existence.
- Quality gates must pass without ML dependencies.
- Coverage must not decrease.
- No classical module may be conditionally imported based on ML availability.
