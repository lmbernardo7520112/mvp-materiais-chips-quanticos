# 1. Executive Verdict

This project employs a **hybrid methodology** that cannot be reduced to any single known framework. It combines Test-Driven Development, Architecture Decision Records, CI/CD quality gates, AI-assisted development with human-in-the-loop governance, scientific risk management, and version-aware policy enforcement into a coherent, evolving system.

**Methodology name:** **AI-RSE GateOps** — AI-assisted Research Software Engineering with Evidence-Gated Scientific Governance.

**Maturity level:** **4.5 / 5** — ADR/TDD/quality-gate governed research software with emerging AI-RSE governance and formal epistemic controls. Not yet level 5 because no experimental validation exists.

**Overall score:** **8.2 / 10** — Strong engineering governance with genuine scientific caution, limited by the absence of experimental data and the inherently simulated nature of council reviews.

**Parecer final:** **Forte** (approaching Muito Forte). The methodology is rigorous, traceable, and scientifically cautious. Its main limitation is not methodological but epistemic: the project governs a model that has not yet been validated against experiment.

# 2. Methodology Name and Classification

## 2.1 Name

**AI-RSE GateOps** — AI-assisted Research Software Engineering with Evidence-Gated Scientific Governance.

## 2.2 One-sentence definition

A human-in-the-loop AI pair-programming methodology where every scope change requires an ADR, every implementation requires TDD RED→GREEN, every release requires CI + quality gates + post-merge validation, and every scientific frontier requires council deliberation before code.

## 2.3 Expanded definition

AI-RSE GateOps is a hybrid development methodology designed for computational science projects where an AI coding agent (Antigravity) acts as the primary implementer under strict human governance. The methodology enforces separation between documentation releases and functional releases, uses version-aware policy files to control what code is authorized at each stage, employs simulated expert councils to deliberate scientific decisions, and tracks every parameter's evidence tier to prevent false calibration claims. The AI agent operates within project-scoped Markdown playbooks (Skills) that codify operational procedures, while the human retains authority over ADR acceptance, merge approval, and scientific interpretation.

# 6. Scientific Governance Assessment

## 6.1 False calibration prevention: **Strong**

The project explicitly separates:
- **Demonstrative mode** (ε=1, proxy ρ_eff, normalized domain) — current state.
- **Literature-scaled mode** (SI constants present but not coupled to solver) — v0.4.4.
- **Physical mode** (Option C, not implemented) — blocked.

`physical_interpretation_allowed=False` is enforced in runtime CSV metadata. This is genuine anti-overclaim infrastructure.

## 6.2 Option B vs Option C separation: **Methodologically correct**

Option B (metadata-only) was implemented in v0.4.4–v0.4.6 without altering numerical outputs. Option C (physical charge mapping) was blocked at every stage by:
- ADR-008 authorizing only Option B.
- Council deliberations recommending against immediate implementation.
- v0.4.8 readiness review identifying 8/10 blocking gaps.
- `option_c_enabled=False` in runtime metadata.

## 6.3 D_it → φ chain risk treatment: **Strong**

The v0.4.8 decision brief explicitly documents gaps in δE_window, sign convention, t_eff, geometry mode, ε substitution, and φ sanity checks. The council voted 0/6 for implementation, 6/6 for ADR-009. This is conservative and scientifically appropriate.

## 6.4 Limitation: No experimental validation

The project has zero comparison with experimental data. All parameters are toy/demonstrative or literature-scaled without device-specific calibration. This is acknowledged but remains the fundamental epistemic gap.

# 10. Strengths

1. **Traceability:** Every change is traceable from ADR → council → PR → commit → tag → validation evidence.
2. **Small releases:** 22 releases in ~7 days. Each is atomic and reversible.
3. **Anti-overclaim:** Multiple layers prevent premature physical interpretation (metadata guards, forbidden terms, policy, council).
4. **Parameter curation:** Evidence tiers (CONST→TX) with literature references prevent uncategorized constants.
5. **Separation of documentation and code releases:** v0.4.1, v0.4.2, v0.4.5, v0.4.7, v0.4.8 are doc-only. v0.4.0, v0.4.4, v0.4.6 are functional.
6. **Version-aware policy:** `policy.json` evolves with the project stage, preventing past-stage violations.
7. **Post-merge validation:** Every release re-validates after merge, not just before.
8. **Tag discipline:** Tags created only after CI green on main. Never premature.
9. **Coverage stability:** 90.86% maintained across 179 tests despite scope expansion.
10. **AI governance infrastructure:** Skills codify operational knowledge for future sessions.

---

# 11. Weaknesses and Risks

1. **Documentation overhead:** ~830 lines of governance documentation added in v0.4.8 alone for zero code changes. Risk of governance-to-code ratio becoming unsustainable.
2. **Simulated councils are not peer review:** Council reports look authoritative but represent AI-generated expert opinions, not independent human validation. Risk of false confidence.
3. **String-based gates:** Quality gates detect forbidden terms but cannot verify dimensional correctness or physical validity. A semantically wrong equation with correct terminology would pass all gates.
4. **No experimental validation:** The entire model is demonstrative. No comparison with measured data exists. Governance rigor does not substitute for empirical validation.
5. **Prompt dependency:** The methodology depends on detailed, well-structured prompts from the human. A vague prompt could bypass governance.
6. **Skills staleness:** Markdown skills have no enforcement mechanism. They advise but do not block. An agent could ignore them.
7. **Report length:** Reports exceed 15 pages per release. Long reports can mask errors through volume.
8. **CI green ≠ scientifically valid:** All tests verify computational contracts, not physical correctness. A solver that converges to the wrong answer would pass all gates.
9. **Model remains toy:** After 22 releases, the core physics is still proxy/demonstrative. The methodology has hardened governance faster than science.
10. **Council consensus bias:** Simulated experts always agree. Real experts would disagree on δE_window, sign conventions, and regularization strategies.

# 14. Recommendations

## 14.1 Short-term (v0.4.9–v0.5)

1. **Create ADR-009** for Option C with specific equations, parameter requirements, and TDD plan.
2. **Add a manufactured solution test** with physical ε to verify solver correctness before Option C.
3. **Reduce report length** — use structured templates that enforce brevity.
4. **Add a "disagreeing expert" role** to councils to test robustness of decisions.

## 14.2 Medium-term (v0.5–v0.6)

5. **Pin dependencies** with `pip-compile` or `uv.lock` for reproducibility.
6. **Add containerization** (Dockerfile) for platform-independent reproduction.
7. **Implement one physical-scale test** (e.g., known Si/SiO₂ capacitor solution) to anchor the model.
8. **Add dimensional analysis tests** that verify units algebraically, not just string labels.

## 14.3 Long-term (v0.7+)

9. **Seek external review** from a human semiconductor physicist.
10. **Compare with TCAD benchmarks** (Sentaurus, COMSOL) for a simple 1D MOS capacitor.
11. **Automate skill enforcement** with pre-commit hooks once skills are stable.
12. **Publish methodology** as a technical note — the AI-RSE GateOps workflow is itself a contribution.

## 15.2 Is this methodology original?

**Yes.** While individual components (TDD, ADR, CI/CD, risk matrices) are well-known, their combination with:
- AI agent as primary implementer under human governance,
- version-aware policy enforcement,
- simulated expert councils for scientific deliberation,
- parameter evidence tiers with anti-overclaim metadata,
- project-scoped Markdown skills as agent governance playbooks,

constitutes a novel workflow that does not have a direct equivalent in the literature. The closest analog is NASA/ESA mission-critical review processes, but adapted for AI-assisted computational science with explicit epistemic humility (marking outputs as demonstrative until experimentally validated).

## 15.3 Final Recommendation

The methodology is **strong and well-suited** for its current stage. The next strategic priority should shift from governance hardening to **scientific content**: implement Option C under ADR-009, compare with known analytical solutions, and begin the path toward experimental validation. The governance infrastructure is now mature enough to support that transition safely.
