# v0.6.1 Process-to-Device Risk Matrix

## Risks

### R1 — Project drifts into bookkeeping-only

- **Severity:** HIGH
- **Failure mode:** Continuous profile/evidence refinement without advancing toward device layer.
- **Mitigation:** Explicit roadmap with release targets and stop conditions. Council review at each milestone.
- **Responsible skill:** scope-guardrails
- **Release gate:** v0.6.2 must be the last pure-C1 release.

### R2 — Project jumps to Schrödinger before C2/Poisson

- **Severity:** CRITICAL
- **Failure mode:** Schrödinger solved in arbitrary potential with no connection to defect chain.
- **Mitigation:** ADR-Schrödinger requires validated Poisson coupling as pre-requisite. Scope-guardrails skill enforces ordering.
- **Responsible skill:** scope-guardrails, physics-dimensional-audit
- **Release gate:** v0.9.0 requires v0.8.0 validated.

### R3 — Maurand anchor misrepresented as calibration target

- **Severity:** CRITICAL
- **Failure mode:** Reports/artifacts imply project reproduces Maurand device.
- **Mitigation:** ADR-011 explicitly states Maurand is device-target anchor only. Report-auditor skill checks all artifacts.
- **Responsible skill:** report-auditor
- **Release gate:** Every release.

### R4 — σ_eff → ρ_eff with unjustified t_eff

- **Severity:** CRITICAL
- **Failure mode:** t_eff chosen arbitrarily, making ρ_eff and all downstream quantities physically meaningless.
- **Mitigation:** ADR-C2 must explicitly define t_eff semantics. Physics-dimensional-audit verifies units.
- **Responsible skill:** physics-dimensional-audit
- **Release gate:** v0.7.0 ADR must resolve t_eff.

### R5 — Poisson potential interpreted physically without context

- **Severity:** CRITICAL
- **Failure mode:** φ(x,y) presented as real device potential without boundary conditions, geometry, or material context.
- **Mitigation:** All φ outputs must carry disclaimers. No physical interpretation without explicit ADR authorization.
- **Responsible skill:** physics-dimensional-audit, report-auditor
- **Release gate:** v0.8.0.

### R6 — Schrödinger prototype mistaken for qubit simulation

- **Severity:** CRITICAL
- **Failure mode:** Eigenstates presented as qubit states without spin, valley, or noise physics.
- **Mitigation:** ADR-002 remains permanent. All Schrödinger outputs must carry "effective confinement prototype" label.
- **Responsible skill:** scope-guardrails, report-auditor
- **Release gate:** v0.9.0.

### R7 — Coherence/fidelity claims without noise/spin model

- **Severity:** CRITICAL
- **Failure mode:** T₁, T₂, or fidelity numbers published.
- **Mitigation:** ADR-002 permanent block. Scope-guardrails skill enforces.
- **Responsible skill:** scope-guardrails
- **Release gate:** Permanent.

### R8 — E1/E2 profile library overinterpreted as experimental validation

- **Severity:** HIGH
- **Failure mode:** Literature-informed profiles presented as experimentally validated for specific devices.
- **Mitigation:** Evidence taxonomy enforces E1/E2 labels. All profiles must carry source attribution and transferability notes.
- **Responsible skill:** report-auditor
- **Release gate:** v0.6.2.

### R9 — Governance overhead delays physics progress

- **Severity:** MEDIUM
- **Failure mode:** Each release requires so much documentation that physics advances stall.
- **Mitigation:** Governance documents should be proportional to risk. Pure-documentation releases can be lighter. Council can streamline if needed.
- **Responsible skill:** ai-rse-gateops
- **Release gate:** Monitor at each milestone.

### R10 — Lack of curated profile library weakens future C2

- **Severity:** HIGH
- **Failure mode:** C2 implemented with only S0 test fixtures, no physically motivated input profiles.
- **Mitigation:** v0.6.2 profile library must precede C2.
- **Responsible skill:** physics-dimensional-audit
- **Release gate:** v0.6.2 must be closed before v0.7.0.
