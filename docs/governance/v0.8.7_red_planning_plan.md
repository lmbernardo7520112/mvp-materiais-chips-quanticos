# v0.8.7 — One-way Poisson Coupling RED Planning Only

## Goal

Plan the future RED test suite for one-way Poisson coupling authorized by ADR-016.

## Authorized

* Define future RED test categories.
* Define expected failure modes.
* Define guardrails.
* Define scope boundaries.
* Define evidence required before future RED.

## Not Authorized

* No tests.
* No implementation.
* No Poisson runtime.
* No solver coupling.
* No physical phi.
* No dummy phi.
* No potential grid.
* No calibration claims.
* No device prediction.
* No paid API.
* No external SDK.
* No goal-like autonomous execution.

## Future RED Categories

The following categories describe what a future RED phase (v0.8.8 or later) would test. No tests are created in v0.8.7.

1. **Import contract for future module** — Verify that a future `one_way_poisson_coupling` module (or equivalent) does not yet exist, triggering `ModuleNotFoundError`.
2. **Source-to-Poisson input adapter contract** — Verify that a future adapter function accepting C3 projected charge density does not yet exist, triggering `ImportError`.
3. **No physical phi metadata** — Verify that no module or output declares `phi_interpretation: physical` or similar physical semantics.
4. **No calibration flags** — Verify that no module sets `calibration_status: calibrated` or claims experimental agreement.
5. **No potential/voltage naming** — Verify that future module does not export symbols named `phi`, `potential`, `voltage`, or `electrostatic_potential` as physical quantities.
6. **Boundary metadata required** — Verify that any future coupling module declares boundary conditions as `demonstrative` and `not_for_physical_interpretation`.
7. **Charge conservation inherited from C3** — Verify that a future coupling must consume only C3-projected charge density and cannot bypass C3 guardrails.
8. **Solver invocation blocked until GREEN** — Verify that calling any future `solve_one_way_poisson` raises an error until the GREEN phase explicitly enables it.
9. **generate_all_results must not emit physical-looking artifacts** — Verify that any future coupling output in `generate_all_results` is clearly flagged as demonstrative.
10. **BudgetOps and Usage Ledger required** — Verify that future RED and GREEN phases must pass BudgetOps and Usage Ledger validation.

## Future RED Expected Failures

These describe conceptual failure modes only. No tests are created.

* `ModuleNotFoundError` for a future `one_way_poisson_coupling` module that does not yet exist.
* `ImportError` for adapter contracts that have not been implemented.
* Guardrail assertion failures if forbidden names (`phi`, `potential`, `voltage`, `electrostatic_potential`) appear as exported physical quantities.
* Metadata validation failure if any output attempts to declare `phi_interpretation: physical`.
* BudgetOps/Usage Ledger failure if paid API, external SDK, or goal-like autonomous execution are required.

## Future RED Prohibited Scope

* No real solver execution.
* No `solve_poisson` or `solve_poisson_2d_demonstrative` invocation from a coupling module.
* No physical interpretation of any numerical output.
* No output of phi/potential/voltage fields as physical quantities.
* No calibration against experimental data.
* No inference of device behavior.
